## Waar de regressie vandaan komt (waarschijnlijk meet-artefact)

**v0.5 heeft ~7 judge-parse failures** (`— — —` scores) vs v0.4's ~3. Elke failure telt mee als score "0" in de mean → drukt het gemiddelde flink omlaag. Patroon uit het rapport:

| Case | v0.4 score | v0.5 score |
|---|---|---|
| (case 1) | 1 / 1 / 0 | — / — / — (judge-parse fail) |
| (case 2) | 3 / 3 / 3 | — / — / — (judge-parse fail) |
| (case 3) | 3 / 3 / 3 | — / — / — (judge-parse fail) |
| (case 4) | 5 / 5 / 5 | — / — / — (judge-parse fail) |

Als die 4 failures hadden gewerkt met scores ~3-5 (zoals v0.4), zou v0.5's gemiddelde naar ~3.5-3.7 stijgen — comparabel met v0.4.

**Waarom v0.5 specifiek?** Twee waarschijnlijke oorzaken:
1. **Langer judge-prompt** (Task 7 voegde 2 metrics `route_correct` + `meta_talk_present` toe) → meer judge-tokens per call → vaker tegen gpt-4o TPM-rate-limit (30k/min).
2. **Langer bot-antwoord bij regenerate** → meer input-tokens voor de judge → idem.

Het rapport toont meerdere `RateLimitError 429` warnings tijdens de run — het script blijft draaien (graceful fallback), maar de judge-call retourneert dan een lege of corrupted JSON die `parseError: true` triggert.

## Mogelijke vervolgacties

1. **Re-run met rate-limit-respectvolle concurrency** — verlaag concurrency van 5 naar 2-3, of voeg een `await sleep(60_000/30 * tokens_estimated)` toe tussen judge-calls. Geeft betrouwbaardere scores zonder bot-veranderingen.
2. **Accepteer als known measurement-artefact** — push v0.5, documenteer dit in de PR, plan een betere eval-infra in v0.6 (mock judge / batch judge / hogere TPM).
3. **Tune claim-regenerate threshold** — als de aanname klopt dat regenerate vaker triggert dan ideaal, zet `claimRegenerateThreshold` van 0.5 naar 0.3 (minder triggers). Eerst meten hoe vaak het feitelijk getriggered is via `query_log.bot_kind` + `extras.claimConfidence`.
