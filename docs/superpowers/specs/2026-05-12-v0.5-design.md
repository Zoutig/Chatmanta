---

## Doel

V0.5 is een **gebundelde kwaliteitsslag** op v0.4. De focus is twee-ledig:

1. **Bot-gedrag dichter bij de gebruiker brengen** — antwoorden mogen menselijker klinken én algemene-kennis-vragen rondom het domein (MKB, SaaS, AI, RAG, klantcontact) niet langer hardhandig afwijzen.
2. **Eval-pipeline betrouwbaar maken** — de parent-excerpt mismatch oplossen zodat we v0.4↔v0.5 eerlijk kunnen vergelijken, en de claim-verificatie die nu alleen meet daadwerkelijk gebruiken om hallucinaties te voorkomen.

Het is bewust een bundel en geen losse PR per item: de items grijpen op elkaar in (general-knowledge en claim-verification raken beide het antwoord-pad; word-ban en general-knowledge raken beide de prompts).

## Items — detail

### Item 1 — General-knowledge router (P2)

**Probleem:** Gebruiker vraagt "Wat zijn MKB-bedrijven?" → bot weigert omdat geen chunk-match boven similarity-threshold 0.4 wordt gevonden. Maar dit is basiskennis die domein-relevant is. Gebruiker verwacht een normale uitleg met disclaimer. Tegelijk wil de gebruiker **niet** dat de bot ineens antwoordt op "schrijf een gedicht over zalmen" of "wat is de hoofdstad van Frankrijk".

**Huidige toestand:**
- Pre-processor (`lib/v0/server/bots.ts:136-164`) classificeert binair: SMALLTALK | SEARCH
- Bij SEARCH + geen chunks ≥ threshold → fallback-pad zonder LLM-call (`lib/v0/server/rag.ts:1003-1017`) → "Dit weet ik niet zeker"

**Voorstel:** Pre-processor uitbreiden naar **vier** categorieën:

```
A) SMALLTALK    — begroetingen, rol-vragen (zoals nu)
B) SEARCH       — in-docs-vraag (zoals nu)
C) GENERAL      — algemene kennis BINNEN domein (NIEUW)
D) OFF_TOPIC    — buiten domein (NIEUW)
```

**Domein-definitie (allowlist):** MKB, SaaS, AI, RAG, chatbots, klantcontact, ondernemerschap, marketing, web-tech, en alles rondom ChatManta-product en Jorion Solutions zelf.

**Flow per categorie:**
- `SMALLTALK` → reply uit pre-process (zoals nu)
- `SEARCH` → retrieve + answer (zoals nu)
- `GENERAL` → skip retrieve, antwoord via aparte "general-knowledge" system-prompt met **verplichte disclaimer**: `"Even kort: dit valt buiten onze specifieke documentatie, maar in het algemeen…"`. Eind met `"Wil je weten hoe ChatManta hier specifiek mee omgaat? Vraag gerust."`
- `OFF_TOPIC` → vaste polite refusal: `"Ik help met vragen rondom ChatManta en aanverwante onderwerpen — denk aan MKB-tech, chatbots, klantcontact. Wat wil je weten?"`

**Output-formaat preProcess uitbreiding:**

```
ACTION: smalltalk
REPLY: <antwoord>

ACTION: search
QUERY: <herschreven vraag>

ACTION: general                ← NIEUW
TOPIC: <het algemene onderwerp, bv. "MKB-bedrijven">

ACTION: off_topic              ← NIEUW
(geen body)
```

**Wijzigingen in code:**
- `lib/v0/server/bots.ts` — v0.5 BotConfig met nieuwe `preProcessSystem` (4-way classificatie) + nieuwe `generalKnowledgeSystem` veld
- `lib/v0/server/rag.ts` — `parsePreProcessOutput()` uitbreiden, nieuwe branches in pipeline (`handleGeneral`, `handleOffTopic`)
- `lib/v0/server/types.ts` — Response-types krijgen optional `category` veld voor eval

**Eval-uitbreiding:** test-set krijgt minstens deze cases:
- `GENERAL`: "Wat zijn MKB-bedrijven?", "Wat is RAG?", "Wat doet een klantcontact-medewerker?", "Wat is SaaS?", "Wat is een vector database?"
- `OFF_TOPIC`: "Wat is de hoofdstad van Frankrijk?", "Schrijf een gedicht over zalmen", "Hoeveel is 743 × 28?"
- Classificatie-accuracy als nieuwe judge-metric (route_correct: yes/no).

**Risk:** Foutclassificatie. "Wat is RAG?" is GENERAL als doc niet beschikbaar, maar SEARCH als wel. Pre-processor kan dit niet weten — hij heeft geen retrieval-info. Mitigatie: laat pre-processor SEARCH kiezen voor alles dat in principe in docs kan staan; alleen bij zero-match in retrieval daarna escaleren naar general-knowledge antwoord. → Aangepaste flow:

```
preProcess → SEARCH chosen → retrieve → IF zero chunks ≥ threshold
                                       → re-classify: GENERAL vs OFF_TOPIC vs FALLBACK
                                       → één korte LLM-call met de allowlist en de vraag
```

Dit is de **definitieve flow** — twee classificatie-stappen, alleen de tweede draait wanneer retrieve faalt. Niet duurder dan vandaag, plus betere UX.

### Item 2 — ParentExcerpt fix (P1 + P4)

**Probleem:** Eval-judge ziet `s.contentExcerpt` (small-chunk, ~240 chars) terwijl de answer-LLM `c.parent_content` (~3200 chars) als context kreeg. Daardoor straft de judge antwoorden die feiten gebruiken uit de parent maar niet uit de small-chunk verschijnen. Resultaat: v0.4 grounding-score is een meet-artefact (zie memory `v04_eval_grounding_artifact.md`).

UI heeft hetzelfde issue: de bronnen-tab toont small-chunk, gebruiker ziet niet wat de LLM werkelijk las.

**Voorstel:**

1. `ChatSource` type krijgt optioneel `parentExcerpt?: string` (`lib/v0/server/types.ts`).
2. `toSource()` (`lib/v0/server/rag.ts:914-924`) zet:
   ```ts
   parentExcerpt: c.parent_content ? c.parent_content.slice(0, 800) : null
   ```
   `contentExcerpt` blijft small-chunk (preciezer voor highlights).
3. `buildJudgeUserPrompt()` (`lib/v0/server/eval.ts:149-153`) prefereert `parentExcerpt` bij weergave aan judge — die ziet nu wat de answer-LLM zag.
4. UI (`app/components/manta/manta-right-panel.tsx` sources-tab): toont parentExcerpt als beschikbaar, met de small-chunk gehighlight als "kern-match".

**Risk:** Response-payload groeit per source met ~560 chars × 5–10 sources = ~5KB extra per chat-response. Acceptabel.

**Niet retroactief:** v0.4 reeds-gelogde queries blijven hun oude (foute) excerpt-data houden. Vergelijking v0.3↔v0.4 met oude rapportages blijft een meet-artefact; v0.4↔v0.5 wordt schoon.

### Item 3 — Claim-verification regenerate (P3)

**Probleem:** v0.4 doet claim-verification (`lib/v0/server/rag.ts:1587-1611`) als telemetrie — ~200ms overhead zonder gedrag-impact. Een antwoord met 0% verifieerbare claims wordt nu gewoon getoond.

**Voorstel:** Bij `verifiedRatio < 0.5` (= meer dan helft van claims niet vector-similar aan enige chunk): **stille regenerate** met striktere system-prompt-variant.

**Stricter prompt-variant** (alleen voor regenerate-pass):
```
Je geeft een tweede poging. Beperk je nu STRIKT tot uitspraken die letterlijk
of bijna letterlijk in de aangeleverde chunks staan. Bij twijfel of een feit
echt in de context staat: laat het feit weg. Liever een korter, voorzichtiger
antwoord dan een antwoord met onverifieerbare claims.
```

**Streaming-implicatie:** dit is het lastige deel. Antwoord wordt nu live gestreamed. Claim-verify gebeurt pas NA `answer-done`. Twee opties:

- **A) Buffer-then-stream:** wacht initial answer af, verify, regenerate indien nodig, stream pas dan naar UI. Verliest streaming UX (~3-5s wachttijd).
- **B) Stream-then-replace:** stream initial direct (gebruiker ziet snel iets), na verify regenerate met `event: replacement` waar UI het oude antwoord vervangt. Behoudt streaming UX maar invasief in UI.

**Beslissing:** Optie **B**, achter een feature-flag `claimRegenerateEnabled: boolean` in BotConfig. Default `true` voor v0.5; bij UX-problemen kunnen we via een hotfix die flag terugzetten zonder de hele versie te muten.

**Limiet:** Max 1 regenerate-poging per query. Anders riskeer je inf loop bij hardnekkig low-verify content.

**UI-werk:**
- Stream-handler in `app/api/v0/chat/route.ts` (of waar SSE-events worden gegenereerd) → nieuw event `event: 'replacement'`, `data: { answer, sources, verification }`
- Chat-message component verwerkt `replacement` door huidige assistant-message volledig te vervangen + een kleine banner: "Antwoord aangepast voor extra zekerheid"

**Cost-impact:** geschat 5-15% van queries triggert regenerate → ~+$0.0005 per query, gemiddeld ~+0.10s p50, +1.5s p95.

**Risk:** Regenerate kan in randgevallen een slechter antwoord geven. Mitigatie: log beide antwoorden + verifiedRatio's in `query_log`; bij eval-regressie analyseren we de regenerate-vergelijking.

### Item 4 — Word-ban soften (P7)

**Probleem:** v0.4 systemPrompt (`lib/v0/server/bots.ts:294-296`) verbiedt expliciet: `"document"`, `"documenten"`, `"documentatie"`, `"bron"`, `"bronnen"`, `"context"`, `"tekst"`, `"informatie"`, `"passage"`, `"uittreksel"`, `"stukje"`. Dat maakt antwoorden soms onnatuurlijk gestileerd — natuurlijke nuance ("Onze documentatie beschrijft…") wordt geblokkeerd.

**Voorstel:** Vervang de hele zwartelijst door:

```
Vermijd meta-talk over je interne bronnen — formuleringen als "volgens de
documentatie", "uit de context blijkt", "in deze passage staat". Schrijf
alsof je het zelf weet. Eén uitzondering: als de gebruiker EXPLICIET naar
herkomst vraagt ("waar lees je dat?", "wat is je bron?"), mag je verwijzen
naar "mijn bronnen".
```

**Risk:** Lichte kans op meta-talk-relapse. Mitigatie: in eval een aparte "meta-talk-presence" check via judge (boolean per antwoord).

### Item 5 — Cascade-kosten centraliseren (P12)

**Probleem:** Hardcoded prijs in `lib/v0/server/rag.ts:1567-1568`:
```ts
const GPT4O_IN = 2.5;
const GPT4O_OUT = 10.0;
```
Bij stack-migratie naar Claude (V1) of een prijswijziging bij OpenAI wordt dit vergeten. Bovendien staat de canonieke cost-tabel al in `lib/ai/llm.ts`.

**Voorstel:** Import `MODEL_COSTS` (of equivalent) uit `lib/ai/llm.ts`, kijk model-naam op runtime op. Cascade-model in BotConfig blijft `cascadeModel: 'gpt-4o'`. Geen gedrag-verandering — alleen onderhoudbaarheid.

### Item 6 — Cache-threshold 0.97 → 0.93 + logging (P9)

**Probleem:** `CACHE_HIT_THRESHOLD = 0.97` (`lib/v0/server/rag.ts:461`) is te streng. Test-set van 17 vragen gaf 0 hits. Cache-write gebeurt wel, dus alle paid-for-storage staat te wachten.

**Voorstel:**
- `CACHE_HIT_THRESHOLD: 0.93` (binnen-bot-variatie; nog steeds "zelfde-vraag-ish").
- Extra logging in `cache_lookup`: bij hit én miss de top-1-sim opslaan in `query_log`. Bij hit kunnen we later analyseren of het antwoord echt past; bij miss kunnen we de optimale threshold bisecten op echte data.
- Bij hit: ook telling van tone/length mismatch loggen (komt vandaag al, blijven loggen).

**Risk:** False hits — twee vragen die 0.93 sim hebben maar inhoudelijk anders. Mitigatie: monitor eval; threshold snel terug naar 0.95 als kwaliteit zakt.

### Item 7 — Followups timeout + error-event (P13)

**Probleem:** `generateFollowUps()` failure logt alleen `console.warn` (`lib/v0/server/rag.ts:1705-1712`). Stream eindigt zonder `followups-done`; UI wacht oneindig op events.

**Voorstel:**
- Wrap followups-call in `Promise.race` met 5s timeout.
- Try/catch: zowel timeout als LLM-fout → emit `event: 'followups-done', data: { followups: [], error: <msg> }`.
- UI hoeft niets te wijzigen — leeg `followups[]` is al een valide state.

## Eval-plan

**Voor PR-merge:** `npm run eval:run-all` op `v0.3, v0.4, v0.5` (alle drie in één run) met:

- **Uitbreiding test-set** met:
  - 5 GENERAL-cases (MKB, RAG, SaaS, klantcontact, ondernemerschap)
  - 3 OFF_TOPIC-cases (hoofdstad, gedicht, rekensom)
  - 5 SEARCH-cases die nu al bestaan (regressie-baseline)
  - 2 multi-turn paren (al-of-niet geslaagd; meet-baseline voor v0.6)

- **Nieuwe judge-metrics**:
  - `route_correct` (boolean) — was de pre-process/fallback-classificatie correct?
  - `meta_talk_present` (boolean) — vervalt de bot in "uit de context blijkt"-stijl?
  - `verified_ratio` (al gelogd) — uitsplitsing per categorie

- **Bestaande metrics** (correctness, grounding) — nu met `parentExcerpt`, dus pas vergelijkbaar v0.4↔v0.5.

**Cache-test:** subset van 5 vragen 2× achter elkaar; meet hit-rate met 0.93 threshold.

**Latency-budget:** v0.5 mag p95 niet meer dan +15% laten lopen vs v0.4 (regenerate is de grote variabele).
