# Hotfix v0.5 — cascade-discipline

**Datum:** 2026-05-13
**Bot-versie:** v0.5 (direct gepatcht — geen v0.6)
**Status:** ontwerp, wacht op review

## Probleem

Korte/abstracte input zoals `"67"` zorgt voor een hallucinatie-pad:

1. Pre-processor rewrite `"67"` → `"Wat is 67?"`.
2. Retrieval haalt één zwakke chunk op (similarity ~0.40-0.50) — net boven de threshold 0.40, dus het zero-hit-pad met `reclassifyAfterZeroHits` (OFF_TOPIC fallback) wordt overgeslagen.
3. gpt-4o-mini antwoordt netjes met `<answer>weet ik niet</answer><confidence>0.2</confidence>`.
4. Cascade-trigger (`confidence < 0.5`) → herdraai met gpt-4o op dezelfde prompt + dezelfde zwakke chunks.
5. gpt-4o vult met priors uit de system-prompt (waar concrete stack-voorbeelden in markdown-vetdruk-instructie staan) en hallucineert "ChatManta gebruikt gpt-4o-mini + pgvector…".

Twee mechanismen veroorzaken dit samen:
- Cascade vertrouwt zelf-gerapporteerde confidence als "probeer harder"-signaal, terwijl het feitelijk een "geen grond"-signaal is.
- De v0.5 systemPrompt bevat concrete ChatManta-feiten als vetdruk-voorbeelden — die fungeren als hallucinatie-bait wanneer chunks weinig houvast bieden.

## Niet-doelen

- Cascade volledig uitschakelen — er zijn cases waar gpt-4o gpt-4o-mini wél nuttig corrigeert mét goede chunks.
- Aparte v0.6 maken — append-only is in deze ronde bewust losgelaten; we patchen v0.5 direct.
- Telemetrie uitbreiden (cascade_used loggen) — wenselijk maar buiten scope.

## Wijzigingen

### B — retrieval-gate vóór cascade

`lib/v0/server/bots.ts`:
- Nieuw optioneel veld op `BotConfig`: `cascadeMinTopSim: number`.
- Default in v0.1: `0` (geen gate). v0.1 en v0.2 hebben `cascadeOnLowConfidence: false` dus het veld doet daar feitelijk niets; v0.3 en v0.4 hebben cascade aan maar erven `0` en blijven dus identiek aan hun huidige gedrag — backwards-compat behouden voor eval-vergelijkingen.
- Op v0.5: `0.50`.

`lib/v0/server/rag.ts` (~regel 1940):
- Extra conditie in de cascade-`if`: `topSim !== null && topSim >= bot.cascadeMinTopSim`.
- `topSim` is al in scope op die regel (gedeclareerd op regel 1541, geüpdatet na selective HyDE).

Effect: bij retrieval met top1 < 0.50 blijft de eerste (gpt-4o-mini) weigering staan. Geen cascade-call naar gpt-4o.

### D — neutrale voorbeelden in v0.5 systemPrompt

`lib/v0/server/bots.ts` (regel ~493-497, binnen de v0.5 systemPrompt-string):

Vervang:
```
• "Onze stack is **OpenAI gpt-4o-mini** voor chat en **pgvector** voor de vector-database."
• "Het pakket kost **€49 per maand** voor het MKB-segment."
• "Het project is opgericht door **Sebastiaan**."
```

Door:
```
• "Onze backend draait op **productnaam** en de database is **technologieX**."
• "Het pakket kost **€XX per maand**."
• "Het project is opgericht door **<naam>**."
```

Effect: de vetdruk-instructie blijft intact, maar er worden geen feitelijke ChatManta-claims meer in de prompt geïnjecteerd die de LLM kan herhalen.

## Bewust uitgesteld (niet in deze hotfix)

**C — lage confidence → reclassify-pad.** Optie eerder besproken: bij `confidence < 0.3` route naar `reclassifyAfterZeroHits` ipv cascade. Reden voor uitstel:
- B alleen al stopt het hallucinatie-pad voor de "67"-case en vergelijkbare zwak-retrieval scenario's.
- C vereist een refactor (zero-hit-blok extracten naar een helper) — niet hotfix-passend.
- Pakken we op als follow-up als telemetrie laat zien dat cascade ondanks B+D nog misgaat.

## Verificatie

- **Unit / manueel:** Stuur `"67"` naar de v0.5 bot via `npm run v0:chat` (of UI). Verifieer dat:
  - cascade NIET triggert (geen extra latency-fase, geen gpt-4o-call in logs).
  - Antwoord is de mini-weigering ("Daar heb ik geen informatie over" of vergelijkbaar).
  - Geen stack-feiten gegenereerd.
- **Eval:** `npm run eval:run-all` op v0.5 voor en na. Verwachting: grounding-metric stabiel of beter; latency licht omlaag (minder cascade-fires); cost licht omlaag.
- **Regressie:** vragen waar cascade nuttig was (sterke retrieval, lage confidence door echte ambiguïteit) — die moeten nog steeds cascaden. top1_sim ≥ 0.50 op die cases is verwacht.

## Bot-versioning-impact

Per `[[v0_is_active_learning_platform]]` geldt normaal "geen mutatie van bestaande v0.X snapshots". Sebastiaan heeft expliciet aangegeven die regel in dit geval te overrulen — de hotfix gaat direct in v0.5. Gevolg: eerdere eval-rapporten over v0.5 zijn niet meer 1:1 reproduceerbaar (prompts en cascade-gedrag wijken). Acceptabel binnen V0-leerplatform-context omdat het een correctheids-fix is, geen experiment.

## Open punten

- Cascade-telemetrie (`cascade_used` in `query_log`) bewust niet meegenomen. Aanrader voor follow-up: toevoegen vóór C zodat de impact van B+D meetbaar is.
